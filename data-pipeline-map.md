# خارطة مسار البيانات — من بطاقة المشروع إلى المحفظة الديناميكية

## الملخص التنفيذي

هذا المستند يرسم المسار الكامل لكيفية وصول الأرقام من مصادرها الأصلية إلى صفحة **محفظة رأس المال الديناميكية** (`CapitalSchedulingPage`). الهدف هو توضيح كل مصدر بيانات، كل تحويل حسابي، وكل نقطة قد يحدث فيها تضارب.

---

## المرحلة 1: مصادر البيانات الأولية (الجداول الأساسية)

المحفظة الديناميكية تستمد أرقامها من **3 جداول رئيسية** + جدول إعدادات اختياري:

| # | الجدول | الدور | من يملأه |
|---|--------|-------|----------|
| 1 | `projects` (بطاقة المشروع) | المصدر الأساسي: الأرض، المساحات، تكلفة البناء، النسب، المدد الزمنية | خزان (من الوثائق) أو المستخدم يدوياً |
| 2 | `marketOverview` (النظرة العامة) | توزيع الوحدات (نسب + متوسط مساحات) لحساب الإيرادات | جول (آلياً) أو المستخدم يدوياً |
| 3 | `competitionPricing` (المنافسة والتسعير) | أسعار البيع (3 سيناريوهات) + خطة السداد + الإيراد المعتمد | جول (آلياً) أو المستخدم يدوياً |
| 4 | `projectCashFlowSettings` (إعدادات التدفق) | تخصيص التوزيع الزمني لكل بند (اختياري — يُستخدم فقط إذا حُفظت إعدادات) | المستخدم من صفحة الإعدادات |

| `gfaOfficesSqft` | GFA مكاتب | حساب المساحة القابلة للبيع + رسوم الفرز |

| `communityFees` | رسوم المجتمع | مبلغ ثابت |
| `surveyorFees` | رسوم المساح | مبلغ ثابت |
| `reraAuditReportFee` | تقرير مدقق ريرا | مبلغ ثابت |
| `reraInspectionReportFee` | تقرير فحص ريرا | مبلغ ثابت |

### من `marketOverview` (النظرة العامة):

| الحقل | الوصف | يُستخدم في |
|--------|-------|------------|
| `residentialStudioPct` + `residentialStudioAvgArea` | نسبة الاستديو + متوسط المساحة | حساب عدد الوحدات → الإيراد |
| `residential1brPct` + `residential1brAvgArea` | نسبة غرفة وصالة + متوسط المساحة | حساب عدد الوحدات → الإيراد |
| `residential2brPct` + `residential2brAvgArea` | نسبة غرفتين + متوسط المساحة | حساب عدد الوحدات → الإيراد |

### من `competitionPricing` (المنافسة والتسعير):

| الحقل | الوصف | يُستخدم في |
|--------|-------|------------|
| `baseStudioPrice`, `base1brPrice`, ... | أسعار السيناريو الأساسي | حساب الإيراد |

| `activeScenario` | أي سيناريو مُفعّل | يحدد أي مجموعة أسعار تُستخدم |
| `approvedRevenue` | الإيراد المعتمد يدوياً | **يتجاوز** الإيراد المحسوب إذا > 0 |

## المرحلة 3: محرك الحساب (`calculateProjectCosts`)

هذه الدالة في `server/investorCashFlow.ts` تأخذ البيانات الثلاثة وتُنتج كائن `ProjectCosts`:

```
المدخلات: project + marketOverview + competitionPricing
                    ↓
calculateProjectCosts(project, mo, cp)
                    ↓
المخرجات: {
  landPrice, agentCommissionLand, landRegistration,
  soilTestFee, topographicSurveyFee, officialBodiesFees,
  designFee, supervisionFee, separationFee,
  constructionCost, communityFees, contingencies,
  developerFee, salesCommission, marketingCost,
  reraUnitRegFee, reraProjectRegFee, developerNocFee,
  escrowAccountFee, bankFees, surveyorFees,
  reraAuditReportFee, reraInspectionReportFee,
  totalRevenue, totalCosts
}
```

### الصيغ الحسابية الرئيسية:

| البند | الصيغة |
|-------|--------|
| تكلفة الإنشاء | `manualBuaSqft × estimatedConstructionPricePerSqft` |
| أتعاب التصاميم | `constructionCost × (designFeePct / 100)` |
| أتعاب الإشراف | `constructionCost × (supervisionFeePct / 100)` |
| رسوم الفرز | `(gfaRes + gfaRet + gfaOff) × separationFeePerSqft` |
| الاحتياطي | `constructionCost × 2%` |
| أتعاب المطور | `totalRevenue × (developerFeePct / 100)` |
| عمولة المبيعات | `totalRevenue × (salesCommissionPct / 100)` |
| التسويق | `totalRevenue × (marketingPct / 100)` |
| تسجيل الأرض | `landPrice × 4%` |
| عمولة وسيط الأرض | `landPrice × (agentCommissionLandPct / 100)` |
| **الإيراد** | `approvedRevenue > 0 ? approvedRevenue : calculatedRevenue` |

### حساب الإيراد المفصل:
```
لكل نوع وحدة (استديو، 1BR، 2BR، 3BR، تجاري صغير/وسط/كبير، مكاتب):
  allocated = saleableArea × (typePct / 100)
  units = floor(allocated / avgArea)
  revenue += avgArea × pricePerSqft × units
```

## المرحلة 4: تعريف البنود الافتراضية (`getDefaultItemDefs`)

بعد حساب التكاليف، يتم تعريف **~25 بند** لكل سيناريو. كل بند يحدد:
- اسمه العربي
- من أي حقل في `costs` يأخذ مبلغه
- مصدر التمويل (مستثمر أو ضمان)
- طريقة التوزيع الزمني (دفعة واحدة / توزيع متساوٍ / مخصص)
- في أي مرحلة يقع

### البنود حسب المرحلة:

| المرحلة | البنود |
|---------|--------|
| **الأرض (paid)** | سعر الأرض، عمولة الوسيط، رسوم التسجيل (4%) |
| **التصاميم (design)** | رسوم حكومية (10%)، فحص التربة، المسح، أتعاب التصاميم، أتعاب المطور-تصاميم (1%) |
| **أوف بلان (offplan)** | رسوم الفرز، تسجيل ريرا، تسجيل الوحدات، NOC، حساب الضمان، رسوم المجتمع، تسويق (25%)، إيداع الضمان (20%)، أتعاب المطور-أوف بلان (1%) |
| **الإنشاء (construction)** | دفعة مقدمة مقاول (10%)، احتياطي (2%)، رسوم بنكية |
| **الإسكرو (escrow)** | رسوم حكومية (90%)، رسوم مجتمع (75%)، دفعات مقاول (70%/20%)، أتعاب المطور-إشراف (3%/2%)، أتعاب الإشراف، المساح، مدقق ريرا، فحص ريرا، تسويق (75%)، عمولة المبيعات |

## المرحلة 5: طبقة الإعدادات المحفوظة (`projectCashFlowSettings`)

هذا الجدول يُمثل **طبقة التخصيص** التي يتحكم فيها المستخدم:

### قاعدة الأولوية:
```
إذا وُجدت إعدادات محفوظة للمشروع + السيناريو:
  → استخدم الإعدادات المحفوظة
  → المبلغ = amountOverride (إذا وُجد) أو computeItemAmountByKey (حساب حي)
  → التوزيع الزمني = حسب distributionMethod المحفوظ
  
إذا لم تُوجد إعدادات:
  → استخدم البنود الافتراضية (getDefaultItemDefs)
  → المبلغ = computeItemAmount (حساب حي من costs)
  → التوزيع = حسب تعريف البند الافتراضي
```

### ما يمكن تخصيصه:

| الإعداد | الوصف |
|---------|-------|
| `amountOverride` | تجاوز المبلغ المحسوب بمبلغ يدوي |
| `distributionMethod` | lump_sum / equal_spread / custom |
| `lumpSumMonth` | شهر الدفعة (للدفعة الواحدة) |
| `startMonth` / `endMonth` | نطاق التوزيع (للتوزيع المتساوي) |
| `customJson` | توزيع مخصص شهر بشهر |
| `fundingSource` | investor / escrow |
| `isActive` | تفعيل/إلغاء البند |

## المرحلة 6: التجميع في المحفظة (`getPortfolioCapitalData`)

هذه الدالة تُنفذ لكل مشروع وتُنتج:

```
لكل مشروع في projects:
  1. اقرأ project + marketOverview + competitionPricing
  2. احسب costs = calculateProjectCosts(...)
  3. احسب الجدول الزمني (phases) من preConMonths, constructionMonths, handoverMonths
  4. اقرأ projectCashFlowSettings (إن وُجدت)
  5. لكل بند:
     - احسب المبلغ (override أو حساب حي)
     - وزّع على الأشهر حسب distributionMethod
  6. اجمع فقط بنود المستثمر (fundingSource = "investor")
     (ما عدا O3 حيث كل البنود على المستثمر)
  7. أنتج: monthlyAmounts[], phaseTotals, grandTotal, paidTotal, upcomingTotal
```

### المخرج النهائي لكل مشروع:

| الحقل | الوصف |
|-------|-------|
| `monthlyAmounts[]` | مصفوفة: المبلغ المطلوب كل شهر |
| `phaseMonthlyAmounts` | تفصيل شهري لكل مرحلة |
| `phaseTotals` | إجمالي كل مرحلة |
| `grandTotal` | الإجمالي الكلي |
| `paidTotal` | ما تم دفعه (أشهر ماضية) |
| `upcomingTotal` | ما هو قادم (أشهر مستقبلية) |
| `itemBreakdown` | تفصيل البنود لكل شهر |

## نقاط التضارب المحتملة

### 1. تضارب المصادر (يدوي vs آلي)

| المشكلة | التفصيل |
|---------|---------|
| **جول يكتب فوق البيانات اليدوية** | عند تشغيل `applyJoelleOutputs`، يُحدّث `marketOverview` و `competitionPricing` مباشرة. إذا كان المستخدم قد عدّل هذه البيانات يدوياً، ستُمحى. |
| **لا يوجد تتبع للمصدر** | لا يوجد حقل يُسجّل "هل هذه القيمة من جول أم يدوية؟" |

### 2. تضارب الإيراد

| المشكلة | التفصيل |
|---------|---------|
| **approvedRevenue يتجاوز كل شيء** | إذا وُجد `approvedRevenue > 0` في `competitionPricing`، يُتجاهل الحساب بالكامل. هذا مقصود لكن قد يُنسى. |
| **activeScenario يغيّر الأرقام** | تغيير السيناريو من "أساسي" إلى "متفائل" يغيّر الإيراد → يغيّر أتعاب المطور + التسويق + عمولة المبيعات |

### 3. تضارب المدد الزمنية

| المشكلة | التفصيل |
|---------|---------|
| **إعدادات محفوظة بأشهر قديمة** | إذا غيّر المستخدم `constructionMonths` في البطاقة بعد حفظ الإعدادات، قد تكون `startMonth/endMonth` المحفوظة خاطئة. الكود يُعالج هذا بـ "remap" لكن فقط لبعض الحالات. |

### 4. تضارب بين الأنظمة المتوازية

| المشكلة | التفصيل |
|---------|---------|
| **جدول `costsCashFlow` منفصل** | يوجد جدول قديم `costs_cash_flow` له حقوله الخاصة (مثل `separationFeePerM2`). هذا الجدول **لا يُغذي** المحفظة الديناميكية مباشرة، لكنه يظهر في صفحات أخرى وقد يُسبب ارتباكاً. |
| **جدول `feasibilityStudies` منفصل** | دراسة الجدوى لها حساباتها المستقلة. لا تُغذي المحفظة مباشرة لكن قد تُعطي أرقاماً مختلفة. |

## ملخص قواعد الأولوية

1. **الإيراد**: `approvedRevenue` (إذا > 0) يتجاوز الحساب التفصيلي
2. **المبلغ لكل بند**: `amountOverride` (إذا محفوظ) يتجاوز الحساب الحي
3. **التوزيع الزمني**: الإعدادات المحفوظة تتجاوز التوزيع الافتراضي
4. **مصدر الأسعار**: `activeScenario` يحدد أي مجموعة أسعار (opt/base/cons)
5. **مصدر توزيع الوحدات**: آخر من كتب في `marketOverview` (جول أو المستخدم)
6. **سيناريو التمويل**: `financingScenario` في بطاقة المشروع يحدد أي بنود تظهر

---

## التوصيات

1. **إضافة حقل "مصدر البيانات"** في `marketOverview` و `competitionPricing` لتتبع هل القيمة من جول أم يدوية
2. **تنبيه عند تشغيل جول** إذا كانت هناك بيانات يدوية ستُكتب فوقها
3. **مراجعة `approvedRevenue`** لكل مشروع — إذا كان > 0 فهو يتجاوز كل حساب
4. **حذف أو أرشفة `costsCashFlow`** لتجنب الارتباك مع النظام الجديد
